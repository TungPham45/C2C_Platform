Current Project Status

The project is now materially closer to a microservices-style setup than it was at the start.

Current database ownership:

- auth_db: User, Address, Wallet, WalletTransaction, Cart
- product_db: Shop, Category, AttributeDefinition, AttributeOption, Product, ProductAttributeValue, ProductVariant, ProductImage, Review
- order_db: Voucher, CartItem, CheckoutSession, ShopOrder, OrderItem
- admin_mod_db: ReportReason, Report
- chat_db: conversations, messages

Resolved Problems

1. Seller context no longer comes from auth-service. DONE.
   Completed changes:
   - auth-service no longer queries `user.shops`
   - JWT no longer carries `shopId`
   - api-gateway no longer forwards `x-shop-id`
   - product-service seller routes now use `x-user-id`
   - product-service resolves seller shop from `product_db`
   - login flow fetches seller context from product-service instead of auth-service

2. Prisma/database alignment after the schema split. DONE.
   Completed changes:
   - all 5 Prisma schemas validate successfully
   - all 5 Prisma clients were regenerated
   - `product_db` and `admin_mod_db` were aligned to their Prisma schemas
   - `auth_db`, `order_db`, and `chat_db` were pulled from the live PostgreSQL structure with `prisma db pull`
   - empty legacy conflict tables were removed from PostgreSQL:
     - `order_db.carts`
     - `chat_db."Conversation"`
     - `chat_db."Message"`
   - `order-client/schema.prisma` now matches the real `order_db`
   - `chat-client/schema.prisma` now matches the real `chat_db`
   - there is no remaining Prisma drift problem for the recent database issue

Open Problems

3. Nx registration for backend services. DONE.
   Completed changes:
   - created `project.json` for:
     - `apps/auth-service`
     - `apps/product-service`
     - `apps/api-gateway`
     - `apps/admin-moderation-service`
   - each backend app is now registered as `projectType: "application"`
   - each backend app now has `build` and `serve` targets
   - Nx project discovery now shows:
     - `auth-service`
     - `product-service`
     - `api-gateway`
     - `admin-moderation-service`
     - `web`

4. Existing data migration. DONE.
   Problem:
   - splitting one system into five databases does not automatically move old production or seed data into the right service database
   Completed changes:
   - created a migration workspace under `scripts/data-migration`
   - added `inventory.cjs` to count rows in all 5 service databases through the generated Prisma clients
   - added `manifest.cjs` with the first reusable migration manifest
   - added auth/product table mapping documentation
   - generated `scripts/data-migration/inventory.latest.json`
   - verified the current local split databases are reachable and contain live local data
   - added executable ETL scripts:
     - `migrate-domain.cjs`
     - `migrate-all.cjs`
   - ETL now supports all 5 domains:
     - auth
     - product
     - order
     - admin moderation
     - chat
   - ETL preserves integer IDs with Prisma upsert and resets Postgres sequences after import
   - verified the packaged npm commands with a dry-run execution path
   Decision:
   - historical backfill from the old shared source is intentionally skipped
   - the current split databases are treated as the active source of truth

5. Service-to-service communication is in progress.
   Problem:
   - the schemas are more microservice-friendly now, but runtime boundaries are not fully implemented yet
   Completed changes:
   - admin-moderation-service no longer needs mock dashboard data for the first admin flow
   - auth-service now exposes `GET /auth/internal/admin/stats`
   - product-service now exposes:
     - `GET /products/internal/admin/stats`
     - `GET /products/internal/admin/pending-shops`
     - `PUT /products/internal/admin/shops/:id/approve`
   - admin-moderation-service now calls auth-service and product-service over HTTP using `fetch`
   - a shared `x-internal-token` / `INTERNAL_SERVICE_TOKEN` check was added for the new internal routes
   - build verification passed for:
     - `auth-service`
     - `product-service`
     - `admin-moderation-service`
   Remaining change:
   - keep cross-service references as IDs or snapshots in the remaining runtime flows
   - add more internal API calls or replicated read models where other services still need external data

6. Some schema-owned models may still need application code.
   Current examples:
   - `WalletTransaction` in auth
   - `CartItem` in order
   - `Review` in product
   Needed change:
   - add service/controller flows only where the application actually needs them

Practical conclusion

The first three targeted problems are finished. The next concrete problem should come from either:

- data migration between the old/shared model and the 5 service databases
- missing service-to-service communication boundaries
- application code for schema-owned models that are not yet used at runtime
